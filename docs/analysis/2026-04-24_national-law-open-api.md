# 🔬 분석: 국가법령정보센터 Open API 사용 전략

## 다루는 문제
laborcase MVP(부당해고·임금체불) 가 **6개 노동 관련 법령의 조문 원문**을 어떤 창구·어떤 주기·어떤 캐시 전략으로 확보해서 판례 매칭·조항 인용에 쓸지를 결정한다. 쿼터·라이선스·개정 이력 추적·인증키 노출 리스크를 모두 만족하는 단일 전략을 정한다.

## 전제 / 가정

- **G1. 데이터 규모**: 6개 법령(근기법·최임법·퇴직급여법·남녀고용평등법·기간제법·파견법) 본문은 합쳐도 수 MB 이내. 벌크 1회 수집이 부담되지 않는다.
- **G2. MVP 사용자 트래픽**: 공개 초기 일일 유니크 방문자 수백~수천 명 가정. 사용자 1명당 평균 조문 뷰 5~10회. → 실시간 API 패스스루 시 하루 수만 요청, 쿼터(10,000/일) 초과 가능.
- **G3. 법적 제약 (CLAUDE.md)**: 잘못된 법 조항 언급 0건이 KPI. 원문은 반드시 출처와 함께 노출. → "지금 시점의 공식 원문" 이 최소 요구이며, 구법 스냅샷은 옵션.
- **G4. 판례와 법령의 관계**: 판례는 **사건 시점의 구법**을 인용한다. 완전한 정합성을 위해서는 `efYd` 기반 시점 스냅샷이 필요하지만, MVP 는 "현행 기준으로 보여주고 개정 고지를 병기" 하는 수준에서도 법적 표현 제약 위반은 아님(단정 표현 없이 "현행 기준으로 해당 조문" 이라고 표기).
- **G5. 배포 인프라**: Cloud Run + Cloud SQL (PostgreSQL 16 + pgvector). Cloud Scheduler + Cloud Run Job 으로 배치 수집 가능.
- **G6. 영문 번역본 범위 밖**: 리서치 조사 제외 범위이며 본 분석도 다루지 않는다.

## 문제 분해

분석 대상을 독립적으로 의사결정 가능한 7개 하위 문제로 나눈다.

1. **P1. 어느 API 창구를 쓸 것인가** — 조문 단위 조회·연혁 등 기능 커버리지와 쿼터가 직결.
2. **P2. 실시간 프록시 vs 사전 수집 캐시** — 쿼터, 응답 지연, 장애 격리 문제.
3. **P3. 저장 스키마 (조문 단위 정규화 vs 원문 XML 통째)** — 판례 매칭·검색 품질과 유지보수 비용의 교환.
4. **P4. 법령 버전 정책 (현행 only vs 시점 스냅샷 포함)** — 판례 정합성 vs 데이터·코드 복잡도.
5. **P5. 개정 감지 방법 (주기 폴링 vs 요청 시 검증)** — 개정 공백 시간과 배치 비용.
6. **P6. 인증키(OC / serviceKey) 관리** — 민감 정보 노출과 레포 분리 전략(후속 분석 노트 `[2026-04-24_public-private-repo-split](./2026-04-24_public-private-repo-split.md)`) 과 맞물림.
7. **P7. 공공누리 출처표시 UI 배치** — 라이선스 준수와 제품 UX 의 균형.

## P1. 어느 API 창구를 쓸 것인가

### 옵션 비교

| 기준 | A. 법제처 직접 (`www.law.go.kr/DRF`) | B. 공공데이터포털 (`apis.data.go.kr/1170000/law`) | C. 병행 사용 |
|---|---|---|---|
| 조문 단위 조회 (`lawjosub`) | ✅ 문서화됨 | ❓ 법령 검색 위주, 조문 단위 미확인 | ✅ |
| 연혁/개정 이력 (`lsHistory`) | ✅ | ❓ | ✅ |
| 판례·해석례 등 부가 대상 | ✅ (`prec`, `expc`, `detc`) | 제한적 | ✅ |
| 쿼터 | 수치 비명시, 사후 제한 | 개발 10,000/일 명시 | 혼합 |
| 인증 | `OC` (이메일 ID + IP 등록) | `serviceKey` | 둘 다 |
| 승인 | 1~2일 | 자동 | — |
| 응답 포맷 | XML/JSON/HTML | XML 위주 | — |
| 운영 예측성 | 중 | 상 | 하 (관리 부담 2배) |
| 유지보수 | 단일 | 단일 | 낮음 (두 창구 동기화 필요) |

### 결정
**옵션 A (법제처 직접) 선택.**

**근거 (리서치):**
- 리서치 기록상 **조문 단위 조회**, **연혁 목록**, **시점 스냅샷(`efYd`)**, **판례·해석례** 등이 모두 DRF 엔드포인트에 문서화됨. 공공데이터포털 경로에서는 조항호목 지원이 공식 확인되지 않음([법제처 국가법령정보 공유서비스 | 공공데이터포털](https://www.data.go.kr/data/15000115/openapi.do)).
- 추후 Phase 2 에서 판례·해석례(`prec`, `expc`) 가 필요한데 이 둘이 같은 DRF 창구에 있어 인증·코드 재사용이 가능.
- 쿼터 수치 미명시는 리스크이나, **벌크 수집 + 로컬 캐시 전략(P2)** 으로 호출 수가 하루 수백 회 수준에 머무르므로 제한 발동 임계치와 거리가 멀다.

**선택하지 않은 이유:**
- **옵션 B**: 조항호목 지원 불확실. MVP 가 조문·항·호 매칭을 핵심으로 하는데 이 기능 없으면 2차 파싱이 필요해 역효과.
- **옵션 C (병행)**: 인증키 2개·엔드포인트 2개·응답 포맷 이종·에러 처리 이중화로 운영 비용이 과하다. 쿼터 분산이라는 이점이 있으나 P2 캐시 전략으로 충분히 회피 가능.

## P2. 실시간 프록시 vs 사전 수집 캐시

### 옵션 비교

| 기준 | A. 요청 시 실시간 프록시 | B. 사전 벌크 수집 + 주기 동기화 | C. 하이브리드 (캐시 미스 시 프록시) |
|---|---|---|---|
| 응답 지연 (p95) | 외부 API 지연 + 네트워크 | DB 조회만 | 캐시 히트 시 DB, 미스 시 프록시 |
| 쿼터 소모 | 방문자 수 비례 (G2 에서 초과 위험) | 법령 수 × 동기화 횟수 (하루 수십) | 캐시 미스 수 |
| 외부 API 장애 영향 | 직접 전파 | 격리 | 부분 격리 |
| 검색 기능 (전문/임베딩) | 불가 (원문 외부에 있음) | 가능 (pgvector 로 색인) | 가능 |
| 라이선스 출처표시 | 쉬움 | 쉬움 (저장 시 메타 동반) | 쉬움 |
| 저장 비용 | 없음 | 수 MB (무시할 수준) | 수 MB |
| 데이터 신선도 | 최신 | 동기화 주기에 종속 (최대 N시간 지연) | 캐시 TTL 만큼 지연 |
| 구현 복잡도 | 낮음 | 중 (배치 + 마이그레이션) | 높음 |

### 결정
**옵션 B (사전 벌크 수집 + 주기 동기화) 선택.**

**근거:**
- MVP 핵심 기능이 **"판례 ↔ 조항 매칭"** 임. 매칭을 위해서는 조항 원문이 **자체 DB 에 색인(임베딩·전문검색)** 되어야 한다. 실시간 프록시는 원문을 DB 에 남기지 않아 매칭 파이프라인과 구조적으로 충돌.
- G3 "잘못된 법 조항 언급 0건" 은 원본 스냅샷과 생성 응답을 교차검증해야 달성 가능. 자체 DB 에 원문이 있어야 검증 훅이 가능.
- pgvector 를 이미 스택에 포함(CLAUDE.md) → 조항 임베딩 저장 경로가 이미 있음.
- 외부 API 장애가 서비스 장애로 전파되지 않음 (가용성 +).

**선택하지 않은 이유:**
- **A**: G2 의 트래픽 가정에서 쿼터 초과·지연·장애 전파 위험이 모두 상존.
- **C**: 두 경로를 모두 유지해야 하므로 구현 복잡도만 추가되고 B 대비 이점이 뚜렷하지 않음. 유일한 장점인 "캐시 미스 시 폴백" 은 개정 직후의 짧은 지연인데, P5 의 **동기화 주기 단축**으로 해결하는 편이 깔끔.

## P3. 저장 스키마

### 옵션 비교

| 기준 | A. 원문 XML 통째 저장 | B. 조문 단위로 정규화 | C. 두 형태 모두 저장 |
|---|---|---|---|
| 조문 매칭 정확도 | 파싱 매번 필요 | 직접 인덱싱 | 직접 인덱싱 |
| 원본 보존 | ✅ | 파싱 오차 위험 | ✅ |
| 임베딩 저장 단위 | 불명확 | 조문 단위가 자연스러움 | 조문 단위 |
| DB 크기 | 작음 | 중 | 약간 큼 |
| 개정 반영 | 전체 교체 | 조문별 diff 가능 | 둘 다 |
| 외부 감사/재현 | 용이 | 원본 대조 어려움 | 용이 |

### 결정
**옵션 C (둘 다 저장) 선택.**

**근거:**
- 원문 XML 은 **법적 감사/재현 목적**으로 불변 보관(raw zone). Cloud Storage `gs://laborcase-raw/law/{lsId}/{lsiSeq}.xml` 수준이면 수 MB 단위라 비용 무의미.
- 정규화 테이블은 검색·매칭·임베딩·UI 렌더링을 담당. `law(id, ls_id, 약칭, ...)`, `law_version(id, law_id, lsi_seq, ef_yd, promulgation_no, ...)`, `article(id, law_version_id, jo, hang, ho, mok, title, body, ef_yd)` 정도.
- G3 를 만족하려면 정규화 테이블 결과를 항상 원문과 대조 가능해야 한다.

**선택하지 않은 이유:**
- **A**: 검색/매칭 때마다 XML 재파싱은 비효율이고 임베딩 단위가 모호해짐.
- **B**: 법적 감사 상황에서 "우리가 저장한 원문" 을 제시할 수 없음. CLAUDE.md 의 "근거 원문 링크 필수" 원칙과 긴장.

## P4. 법령 버전 정책 — **사용자 결정 필요**

### 옵션 비교

| 기준 | A. 현행 only | B. 현행 + 시점 스냅샷 |
|---|---|---|
| MVP 구현 복잡도 | 낮음 | 중 (`law_version` 다수 + 판례 시점 조인) |
| 판례 정합성 | 낮음 (옛날 판례는 현행과 조문 번호가 다를 수 있음) | 높음 |
| UX 부담 | 낮음 | "이 판례 시점의 구법은 다음과 같습니다" 라벨 필요 |
| CLAUDE.md 법적 제약 위반 여부 | 위반 아님 (현행 기준임을 명시하면) | 위반 아님 |
| 데이터 크기 | 수 MB | 수십 MB (모든 개정본) |
| 구현 시점 | 즉시 | 추후 단계적 확장 가능 |

### 결정 (잠정)
**옵션 A (현행 only) 로 MVP 출시 → Phase 2 에서 시점 스냅샷 추가.**

**근거:**
- G4 에 따라 "현행 기준" 을 UI 에 명시하면 법적 표현 제약 위반 아님.
- MVP 의 부당해고·임금체불은 근로기준법 주요 조문 번호가 **최근 10년 내 안정적**이라 판례-조문 번호 불일치 빈도가 높지 않다고 추정 (검증 필요).
- 스키마는 처음부터 `law_version` 분리하여 설계 → Phase 2 에서 row 만 추가하면 됨 (전제: P3 의 3-테이블 구조).

**❓ 사용자 결정 요청:**
판례가 참조한 조문 번호가 개정으로 이동된 경우 MVP 에서 어떻게 보여줄 것인가?
- **(a)** 현행만 제시 + "원문 링크 참고" 로 우회 → 빠른 MVP, 일부 판례에서 UX 혼란 가능.
- **(b)** 구법 스냅샷까지 수집해 "판례 당시 조문" 라벨로 병기 → 정합성 상승, 구현 +1~2주.

> [!question] 이 항목은 Zephyr 결정 후 확정한다. 잠정적으로 (a)로 가정하고 플랜 단계로 넘어간다.

## P5. 개정 감지 방법

### 옵션 비교

| 기준 | A. 매 요청 시 `lsiSeq` 비교 | B. 일 1회 배치 (`lsHistory` 폴링) | C. 시간 1회 배치 |
|---|---|---|---|
| 쿼터 소모 | 요청 수 비례 (G2 위험) | 하루 6회 (6개 법령) | 하루 144회 |
| 개정 반영 지연 | 실시간 | 최대 24시간 | 최대 1시간 |
| 구현 복잡도 | 높음 | 낮음 | 낮음 |
| CLAUDE.md "잘못된 법 조항 0건" 위험 | 낮음 | 낮음 (개정은 사전 공포 → 시행일 예고가 일반적) | 매우 낮음 |

### 결정
**옵션 B (일 1회 배치) 선택. 개정 임박(시행일 7일 전) 법령은 주기를 1시간으로 단축.**

**근거:**
- 법령 개정은 공포→시행일 사이 **최소 유예기간이 법제도적으로 존재**(부칙 시행일 조항)하여 수 시간 단위 동기화로 충분.
- `lsHistory` 응답에는 공포일·시행일이 들어오므로, 결과를 본 뒤 "시행일 ≤ 오늘+7일" 인 대상만 시간 단위로 재폴링하는 2단 스케줄러가 단순하고 쿼터도 아낀다.
- Cloud Scheduler + Cloud Run Job 조합이 이미 스택 내에 있음.

**선택하지 않은 이유:**
- **A**: P2 의 캐시 전략(사전 수집)과 모순. 매 요청시 비교는 실시간 프록시와 같은 트래픽 패턴.
- **C**: 정상적인 개정 패턴 대비 과잉. 야간 Cloud Run 콜드스타트 비용만 증가.

## P6. 인증키(OC / serviceKey) 관리

### 결정
**Google Secret Manager 저장 + Cloud Run 서비스계정 권한으로 런타임 주입. 레포에 키 텍스트 금지.**

**근거:**
- CLAUDE.md 원칙 4번 "민감 정보를 코드에 직접 쓰지 않음" 에 부합.
- OC 파라미터는 URL 쿼리스트링으로 나가는 구조라 **서버 액세스 로그·외부 로그 수집기에 평문 노출** 위험. 백엔드 API 서버에서만 호출하고 응답은 자체 DB 로 덮어 프론트/클라이언트 로그에 절대 노출하지 않는다.
- pre-commit hook 으로 `gitleaks` 추가 (CLAUDE.md 의 repo 분리 후속 분석에서도 동일 체크리스트 명시).
- OC 와 IP 등록이 묶여 있어, Cloud Run 의 Egress 를 고정 IP(Cloud NAT) 로 내보내 법제처에 등록해야 함. → 인프라 요건으로 플랜에 반영.

**반례 검토:**
- "Cloud Run 서비스계정 만으로는 Secret 을 가져오지 못한다" → `secretmanager.secretAccessor` 역할 부여 + `--set-secrets` 로 환경변수 매핑하면 해결. 애플리케이션 코드는 `OC` 환경변수만 읽으면 됨.

## P7. 공공누리 출처표시 UI 배치

### 결정
**3곳 동시 표기.**
1. **조문 상세 페이지 상단**: "출처: 법제처 국가법령정보센터 · 공공누리 제1유형 · [원문 보기](https://www.law.go.kr/lsInfoP.do?lsiSeq=...)"
2. **사이트 footer**: "본 서비스는 법제처 국가법령정보(공공누리 제1유형)를 이용합니다."
3. **API 응답 메타**: `source.license=KOGL-1`, `source.url=...`, `source.retrieved_at=ISO8601` 필드를 조문 API 응답에 포함 → B2B 또는 외부 확인 요청 시 근거 제공.

**근거:**
- 리서치: 공공누리 1유형은 "온라인 표시 가능한 경우 출처 하이퍼링크 필수". 페이지 하나에만 두는 것보다 **조문 단위** 로 링크가 붙어야 사용자가 원문 재확인이 용이.
- CLAUDE.md 의 "모든 결과 페이지 하단에 디스클레이머 고정 표시" 와 함께 배치. 디스클레이머 ≠ 출처표시 이므로 별개 영역으로 구분.
- "특수관계처럼 오인되는 표시 금지" 조항 위반 회피: 표기에 "법제처 공식 파트너" 같은 문구 금지를 설계 가이드에 명시.
