# DRF Open API 실제 응답 스키마 메모

플랜 §R8 ("응답 스키마가 문서와 다를 수 있음") 검증용. Task 0 PoC 에서 실제 호출로 얻은 **검증된 스키마와 제약**만 기록.

- **OC**: 개인 개발계정 사용 (2026-04-24 승인). 값은 Secret Manager / 로컬 환경변수 `OC_LAW` 로만 주입, 레포에 커밋 금지.
- **fixture 경로**: `docs/research/drf-fixtures/*.xml` (OC 는 placeholder `TESTOC` 로 치환되어 있음).

### lawjosub (조항호목 조회)

- 기본 구조는 lawService (target=law) 의 `<조문>` 부분만 떼어낸 형태.
- `<항>`, `<호>`, `<목>` 까지 중첩. 문서 설명과 일치.
- `JO` 파라미터 포맷: **숫자 그대로 4자리 + "00" + 6자리 zero-pad**. 예시 제23조 → `002300`.
- fixture: `drf-fixtures/lawjosub_근로기준법_23조.xml`.

## 📌 discovered: JO 파라미터 포맷 정정

리서치 문서에 "6자리 zero-pad" 로 기록되어 있으나 실제는 **"실제 번호 * 100 을 6자리 zero-pad"** 로 전송해야 한다.

| 조문 | 파라미터 값 |
|---|---|
| 제3조 | `000300` |
| 제23조 | `002300` |
| 제100조 | `010000` |

`LawOpenApiClient` (Task 3) 에서 `locator → param` 변환을 정확히 구현할 것. 단위 테스트 필수.

## 📌 discovered: 응답 인코딩

- 응답의 XML 선언은 `<?xml version="1.0" encoding="utf-8"?>` (소문자 "utf-8").
- `lawSearch` 는 **UTF-8** (대문자), `lawService`/`lawjosub` 은 **utf-8** (소문자). JVM `XmlMapper` 는 둘 다 처리 가능하지만 스냅샷 비교 시 혼동하지 말 것.

## 📌 discovered: CDATA 내 전각 기호

- 항번호 CDATA 값에 **전각 숫자 괄호** (`① `, `② ` ...) + 후행 공백 1개 포함.
- 항내용도 선행 공백 2칸 + `① ` 이 포함된 형태로 내려온다.
- 파서에서 `.trim()` 처리 시 `항번호` 는 `①`, 항내용은 실제 문장만 남도록 전처리 필요.

## 📌 discovered: 검색 결과의 1순위가 반드시 본법이 아님

- `query=근로자퇴직급여 보장법` 검색 결과 1순위가 본법이었으나, `query=남녀고용평등과 일ㆍ가정 양립 지원에 관한 법률` 의 첫 결과는 "타법개정" 이었음. 대부분의 경우 1순위 = 현행 본법이지만, 검색 결과를 **맹신하지 말고** `법령명한글` 과 `현행연혁코드=현행` 을 함께 확인해 본법을 선별해야 한다.
- Task 6 FullSyncJob 에서 lsId 는 **labor-law-identifiers.md 의 확정 값**으로 하드코딩하고 검색 결과에 의존하지 않는다.

## 📌 discovered: `법령상세링크` 는 OC 포함

- `<법령상세링크>` 값에 `?OC=TESTOC&...` 가 이미 들어있다. 응답을 가공해 **OC 쿼리를 제거**하고 사용자에게 보여줄 공식 URL(`https://www.law.go.kr/lsInfoP.do?lsiSeq=...`) 로 다시 만들어야 한다.
- Task 9 `SourceMetaFactory` 에 `rewriteUrl()` 단계 포함.

## 호출 예상 트래픽 (재추정)

Task 0 의 실측 사이즈로 다시 산정:

- 초기 벌크(6개 법령 × 본문 1회): 6 호출 × ≈260KB = 1.5MB 다운로드. 1분 이내.
- 일일 동기화 (6개 법령 × 검색 1회): 6 호출 × 2.6KB = 16KB. 매우 가볍다.
- 임박 법령 시간 폴링(최악 6개 × 24시간 = 144 호출/일). 쿼터 우려 없음.

→ 분석 §R1 (쿼터 우려) 은 실측상 **현실성 매우 낮음**. 다만 남용 방지 원칙은 유지.
